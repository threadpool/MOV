Contract Overview

The MerchantOfVenice contract implements a secured lending escrow system where:

  • Shylock (the lender) sets the loan terms when deploying the contract
  • Antonio (the borrower) deposits collateral worth twice the loan amount
  • Shylock then sends the loan directly to Antonio
  • Antonio must repay before the deadline to recover his collateral
  • If Antonio defaults, Shylock claims the full collateral

Core Design Principle

Every ETH that enters the contract has a defined exit path. No funds can ever be stranded. This is achieved through careful state management and enforced sequencing via modifiers.

License and Pragma Declaration

Line 1 This comment specifies the software license under which this code is released. The SPDX (Software Package Data Exchange) standard provides a consistent way to declare licenses. MIT is a permissive open source license that allows anyone to use, modify, and distribute this code.

Line 2 The pragma directive tells the Solidity compiler which version to use. The caret symbol ^ means "this version or any newer minor version but not the next major version." So ^0.8.20 accepts 0.8.20, 0.8.21, 0.8.22, etc., but not 0.9.0.

Why this matters: Solidity evolves rapidly, and different versions can have breaking changes. This declaration ensures your code compiles correctly.

Parties (State Variables)

What Are State Variables?

State variables are permanently stored on the blockchain. They persist between function calls and cost gas to write. Think of them as the contract's permanent memory.

Line 33 Declares a state variable named shylock of type address with public visibility.

  • address is an Ethereum wallet address (20 bytes, e.g., 0x1234...)
  • public means Solidity automatically creates a getter function so anyone can read this value
  • This stores the lender's wallet address
address public antonio;

Line 36 Stores the borrower's wallet address, also publicly readable. Same structure as the shylock variable above.

  • uint256 means a whole number from 0 to 2256-1 (very large)
  • 1 ETH = 1018 wei
  • public again means readable by anyone

Deadlines and Time Windows

How Blockchain Time Works

Solidity uses Unix timestamps (seconds since January 1, 1970). The current blockchain time is available via block.timestamp. Deadlines are stored as Unix timestamps.

uint256 public depositDeadline;

Line 58 Stores the Unix timestamp by which Antonio must deposit collateral. Set when the contract is deployed. After this time, Antonio can no longer deposit.

uint256 public disbursementDeadline;

Line 62 Stores when Shylock must send the loan. This is set when Antonio deposits collateral. If Shylock misses this deadline, Antonio can reclaim his collateral.

uint256 public repaymentDeadline;

Line 66 Stores when Antonio must repay the loan. Set when Shylock disburses the loan. If Antonio misses this deadline, Shylock can claim the collateral.

uint256 public depositWindowSeconds;
uint256 public disbursementWindowSeconds;
uint256 public repaymentWindowSeconds;

Lines 74-76 These store the duration of each window in seconds. Unlike the deadline timestamps above, these store durations (how long, not when). They are set at deployment and used to calculate the actual deadlines.

State Flags

What Are Boolean Flags?

Boolean variables store either true or false. They track whether something has happened. In Solidity, booleans default to false if not explicitly set.

bool public collateralDeposited;

Line 84 Tracks whether Antonio has deposited the required collateral. Starts as false, becomes true when Antonio calls depositCollateral().

bool public disbursed;

Line 87 Tracks whether Shylock has sent the loan to Antonio. Starts as false, becomes true when Shylock calls disburseLoan().

bool public settled;

Line 91 Tracks whether the contract has reached its final state. Once true, no further functions can be called. This is the "circuit breaker" that prevents any action after the contract completes.

Reentrancy Guard

Understanding Reentrancy Attacks

A reentrancy attack happens when a malicious contract calls back into your contract before the first call finishes. This can allow attackers to drain funds. The reentrancy guard prevents this.

bool private _locked;

Line 98 A private boolean flag used for the reentrancy guard. private means only this contract can access it (unlike public). The underscore prefix is a naming convention for internal variables.

modifier nonReentrant() {
require(!_locked, "Reentrant call");
_locked = true;
_;
_locked = false;
}

Lines 100-105 This is a modifier that prevents reentrancy.

Breaking it down:

  1. require(!_locked, "Reentrant call"); → If already locked, reject the call with error message
  2. _locked = true; → Set the lock before executing the function
  3. _; → THE UNDERSCORE is where the actual function body runs (more below)
  4. _locked = false; → Release the lock after function completes

The Underscore _ Explained

The underscore _; is a special placeholder in modifiers. It represents "execute the function body here." When you attach a modifier to a function, Solidity inserts the function's code at the position of the underscore.

Example:

function myFunction() public nonReentrant {
    // function body
}

This is equivalent to:

function myFunction() public {
    require(!_locked, "Reentrant call");
    _locked = true;
    
    // function body runs here
    
    _locked = false;
}

The modifier "wraps" your function code with setup (before _) and cleanup (after _).

Events

What Are Events?

Events are Solidity's way of logging information that external applications can listen to. When you emit an event, it is recorded in the blockchain's transaction log. Frontend applications, block explorers, and indexers can watch for these events.

Key point: Events do not cost much gas and are not accessible from within smart contracts. They are for off-chain consumption only.

event ContractCreated(
address indexed shylock,
address indexed antonio,
uint256 loanAmount,
uint256 collateralAmount,
uint256 depositDeadline
);

Lines 113-119 Declares an event that will be emitted when the contract is deployed.

Breaking down the syntax:

  • event → keyword to declare an event
  • ContractCreated → the event name (convention: PascalCase)
  • The parentheses contain the parameters that will be logged
  • indexed → makes these parameters searchable (you can filter events by indexed values). Maximum 3 indexed parameters per event.
event CollateralDeposited(uint256 amount, uint256 disbursementDeadline);

Line 122 Event emitted when Antonio deposits collateral. Records the amount and the new disbursement deadline. No indexed parameters here, so you cannot efficiently search by these values.

event LoanDisbursed(uint256 amount, uint256 repaymentDeadline);

Line 125 Event emitted when Shylock sends the loan. Records the loan amount and repayment deadline.

event LoanRepaid(uint256 repaymentAmount, uint256 collateralReturned);

Line 128 Event emitted when Antonio successfully repays. Records both the repayment and the collateral being returned.

event CollateralReclaimed(uint256 amount);

Line 131 Event emitted when Antonio reclaims collateral because Shylock failed to disburse on time.

event BondForfeited(uint256 amount);

Line 134 Event emitted when Shylock claims the collateral because Antonio defaulted.

event ContractCancelled();

Line 137 Event emitted when Shylock cancels because Antonio never deposited collateral. No parameters because nothing interesting happened.

How emit Works

Later in the contract, you will see lines like:

emit ContractCreated(msg.sender, _antonio, _loanAmount, ...);

The emit keyword triggers the event and writes it to the blockchain logs. The values in parentheses must match the event declaration exactly (same types, same order).

Modifiers and Access Control

What Is a Modifier?

A modifier is reusable code that runs before (and optionally after) a function. It is like a "wrapper" around your function. Modifiers are used for access control, state validation, and preconditions.

Remember the underscore _; represents where the actual function code runs.

Access Control Modifiers

modifier onlyShylock() {
require(msg.sender == shylock, "Only Shylock may call this");
_;
}

Lines 145-148 Restricts function access to Shylock only.

Breaking it down:

  • msg.sender → the wallet address that called this function
  • require(condition, "error message") → if the condition is false, revert the transaction with the error message
  • If msg.sender == shylock is true, execution proceeds to _; where the function body runs
  • If false, the transaction reverts immediately with "Only Shylock may call this"
modifier onlyAntonio() {
require(msg.sender == antonio, "Only Antonio may call this");
_;
}

Lines 150-153 Same pattern as onlyShylock, but checks for Antonio's address. Only Antonio can call functions with this modifier.

}

Lines 192-195 Opposite of beforeDepositDeadline. Ensures the function is called after the deadline has passed.

_;
}

require(block.timestamp > disbursementDeadline, "Disbursement window still open");
_;
}

Lines 202-205 Function must be called after disbursement deadline passes.

require(block.timestamp <= repaymentDeadline, "Repayment window has closed");
_;
}

require(block.timestamp > repaymentDeadline, "Repayment window not yet closed");

Stacking Modifiers

Functions can have multiple modifiers. They execute in order from top to bottom. For example:

function myFunction()
    external
    onlyAntonio
    notSettled
    beforeDeadline
{
    // function body
}

This function checks: (1) caller is Antonio, (2) contract not settled, (3) before deadline, then runs the function body.

The constructor is a special function that runs exactly once when the contract is deployed to the blockchain. It initializes the contract's state. After deployment, the constructor can never be called again.

  • Takes 5 parameters: Antonio's address and four timing values
  • Parameters prefixed with underscore (_antonio, _loanAmount, etc.) is a naming convention to distinguish them from state variables

Core Functions

Safe ETH Transfer Helper

function _safeTransfer(address to, uint256 value) internal {
(bool success, ) = payable(to).call{value: value}("");
require(success, "ETH transfer failed");
}

Lines 268-271 A helper function to safely send ETH.

Breaking down each part:

  • internal → only this contract can call this function
  • payable(to) → converts the address to a payable address (can receive ETH)
  • .call{value: value}("") → low level call that sends value wei to address to
  • (bool success, ) → the call returns two values; we capture success and ignore the second with a blank
  • require(success, ...) → if the transfer failed, revert the entire transaction

Why not use .transfer()? Older Solidity used address.transfer() but it only forwards 2300 gas, which can fail with smart contract wallets. This method forwards all available gas and is safer.

View Functions

What Are View Functions?

Functions marked view promise not to modify any state. They only read data and return it. Calling view functions is free (no gas cost) when called externally because they don't create transactions.

function timeToDepositDeadline() external view returns (uint256) {
if (block.timestamp >= depositDeadline) return 0;
return depositDeadline - block.timestamp;
}

Lines 400-403 Returns how many seconds remain until the deposit deadline.

Breaking down the logic:

  • external view → can be called from outside, reads only
  • returns (uint256) → specifies this function returns an unsigned integer
  • if (block.timestamp >= depositDeadline) → if deadline has passed
  • return 0; → return 0 seconds remaining
  • return depositDeadline - block.timestamp; → otherwise subtract current time from deadline
function timeToDisbursementDeadline() external view returns (uint256) {
if (!collateralDeposited) return 0;
if (block.timestamp >= disbursementDeadline) return 0;
return disbursementDeadline - block.timestamp;
}

Lines 407-411 Returns seconds until disbursement deadline.

  • Returns 0 if collateral was never deposited (deadline never set)
  • Returns 0 if deadline has passed
  • Otherwise returns seconds remaining
function timeToRepaymentDeadline() external view returns (uint256) {
if (!disbursed) return 0;
if (block.timestamp >= repaymentDeadline) return 0;
return repaymentDeadline - block.timestamp;
}

Lines 415-419 Returns seconds until repayment deadline.

  • Returns 0 if loan was never disbursed (deadline never set)
  • Returns 0 if deadline has passed
  • Otherwise returns seconds remaining
function getBalance() external view returns (uint256) {
return address(this).balance;
}

Lines 423-425 Returns the contract's current ETH balance in wei.

  • address(this) → the address of this contract itself
  • .balance → built in property that returns the ETH balance

Useful for verifying the contract holds the expected collateral amount.

Summary: Key Solidity Concepts Covered

  1. State variables → permanent storage on blockchain (address, uint256, bool)
  2. Modifiers → reusable code wrappers with the underscore (_) placeholder
  3. Events → logging mechanism using emit keyword
  4. require() → input validation that reverts on failure
  5. msg.sender → the address calling the function
  6. msg.value → the amount of ETH sent with the call
  7. block.timestamp → current blockchain time
  8. payable → allows functions and addresses to receive ETH
  9. external vs internal → who can call the function
  10. view → read only functions that cost no gas externally

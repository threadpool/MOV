The Merchant of Venice

A plain-English walkthrough of a Solidity smart contract
Pictet Geneva Crypto Club · April 21 2026

What is this contract?

This document walks through MerchantOfVenice a fully on-chain secured lending escrow line by line. No programming experience is assumed. Every concept is explained in plain English before the code is shown.

The contract models the Shakespeare story as a complete, trustless lending agreement. Antonio deposits collateral worth twice the loan before receiving anything. Shylock then disburses the loan on-chain. If Antonio repays, he recovers his full collateral. If he defaults, Shylock claims it. Every step is enforced by code no bank, no lawyer, no trust required at any stage.

🏦

Deploy

Shylock sets the terms. No ETH locked yet. Three time windows established.

🔐

Collateral

Antonio deposits 2× the loan. Locked in the contract before a penny is lent.

💸

Lend

Shylock disburses the loan on-chain to Antonio. Repayment clock starts.

⚖️

Enforce

Repay and recover collateral or default and forfeit it. Code decides.

The contract declaration

Parties and financial terms

address public shylock;          // The lender deploys the contract
address public antonio;          // The borrower

uint256 public loanAmount;       // The loan, in wei
uint256 public collateralAmount; // Always loanAmount * 2, computed automaticallySolidity

address is an Ethereum wallet a unique 42-character identifier beginning with 0x. uint256 is a non-negative whole number. All ETH values are stored in wei the smallest unit of Ethereum (1 ETH = 10¹⁸ wei) to eliminate any possibility of rounding errors in arithmetic.

💡 Why 200% collateral? Antonio must deposit twice what he borrows. If he repays, he recovers all of it net position zero. If he defaults, Shylock recovers the loan amount plus the same again as a penalty. The asymmetry is the deterrent. This is the pound of flesh, expressed in ETH.

collateralAmount is never entered by anyone it is computed automatically as loanAmount * 2 in the constructor. No human can set it to a different value.

4

The three deadlines

uint256 public depositDeadline;      // Set at deployment
uint256 public disbursementDeadline; // Set when Antonio deposits
uint256 public repaymentDeadline;    // Set when Shylock disbursesSolidity

These are Unix timestamps seconds since 1 January 1970 enforced by the Ethereum blockchain clock, not a server or calendar. The critical design principle is that each deadline is set progressively, triggered only by the previous action:

📅

depositDeadline

Set at deployment. Antonio must deposit collateral before this or the contract can be cancelled.

📅

disbursementDeadline

Set when Antonio deposits. Shylock must disburse within this window or Antonio can reclaim.

📅

repaymentDeadline

Set when Shylock disburses. Antonio's clock only starts when he actually has the money.

✓ Why progressive deadlines matter: If the repayment deadline were set at deployment, it would start counting before Antonio had received a penny deeply unfair. By setting it at the moment of disbursement, Antonio's window only opens when the loan arrives in his wallet. Both parties get their fair and enforced time.

State flags

bool public collateralDeposited; // true once Antonio deposits
bool public disbursed;           // true once Shylock disburses
bool public settled;             // true once contract is finally resolvedSolidity

Three boolean flags track where the contract is in its lifecycle. They can only move in one direction from false to true and together they enforce the correct sequence of events.

FlagSet byWhat it unlocksWhat it blocks
collateralDepositeddepositCollateral()disburseLoan() becomes callableA second deposit
disburseddisburseLoan()repay() and claimBond() become callablereclaimCollateral()
settledAny terminal functionNothing contract is permanently closedEvery function via notSettled
💡 The sequencing guarantee: Because claimBond() requires disbursed == true, Shylock cannot claim a default on a loan he never made. Because repay() requires disbursed == true, Antonio cannot repay a loan he never received. The flags enforce correct ordering no shortcuts, no skipping steps.

The reentrancy guard

bool private _locked;

modifier nonReentrant() {
    require(!_locked, "Reentrant call");
    _locked = true;
    _;
    _locked = false;
}Solidity

Works like a door bolt. Before any ETH-moving function runs it checks the bolt is open, locks it, executes the function, then unlocks it. If a malicious contract attempts to call back in before the function finishes the bolt is already locked and the attempt is immediately rejected.

⚠️ The 2016 DAO hack: A contract without this guard was drained of $60 million. An attacker called the withdrawal function repeatedly before the contract had recorded that the first withdrawal occurred. This guard prevents exactly that attack. Applied to every function that moves ETH in this contract.

Modifiers access, state and time

// Access control
modifier onlyShylock()              { require(msg.sender == shylock, ...); _; }
modifier onlyAntonio()              { require(msg.sender == antonio, ...); _; }
// State control
modifier notSettled()               { require(!settled, ...); _; }
modifier hasCollateral()            { require(collateralDeposited, ...); _; }
modifier notDisbursed()             { require(!disbursed, ...); _; }
modifier isDisbursed()              { require(disbursed, ...); _; }
// Time control
modifier beforeDepositDeadline()    { ... }
modifier afterDepositDeadline()     { ... }
modifier beforeDisbursementDeadline(){ ... }
modifier afterDisbursementDeadline() { ... }
modifier beforeRepaymentDeadline()  { ... }
modifier afterRepaymentDeadline()   { ... }Solidity

Modifiers are reusable rules bolted onto functions. Each one runs at the moment the function is called not at deployment. If any check fails the entire transaction reverts instantly and no state changes. Twelve modifiers in three categories give the contract precise, layered control over who can do what and when.

The constructor

constructor(
    address _antonio,
    uint256 _loanAmount,
    uint256 _depositWindowSeconds,
    uint256 _disbursementWindowSeconds,
    uint256 _repaymentWindowSeconds
) {
    shylock          = msg.sender;
    antonio          = _antonio;
    loanAmount       = _loanAmount;
    collateralAmount = _loanAmount * 2;  // 200% hardcoded
    depositDeadline  = block.timestamp + _depositWindowSeconds;
    // disbursementDeadline and repaymentDeadline set progressively later
    emit ContractCreated(...);
}Solidity

Runs exactly once at deployment, never again. Takes five parameters and establishes the entire agreement. msg.sender is whoever deployed Shylock. block.timestamp is the blockchain clock in Unix seconds. No ETH is locked at this stage deployment is lightweight. Only the deposit window starts counting immediately.

✓ Notice: collateralAmount = loanAmount * 2 is computed and written to the blockchain permanently at this moment. Antonio cannot deposit less. Nobody can change it. The 200% requirement is mathematically locked into the agreement from the instant it is created.
10

The safe transfer helper

function _safeTransfer(address to, uint256 value) internal {
    (bool success, ) = payable(to).call{value: value}("");
    require(success, "ETH transfer failed");
}Solidity

internal means only this contract can call it it is a private helper invisible to the outside world. Used by every function that sends ETH. The .call{value:v}("") pattern forwards all available gas rather than the fixed 2,300-unit limit of the old .transfer() method, which could fail if the recipient is a smart contract with any logic. The require(success) ensures no transfer can fail silently.

11

The six action functions

These are the only ways the contract's state or ETH balances can change after deployment. Each is guarded by a precise combination of modifiers.

depositCollateral()
Antonio locks twice the loan amount starts the disbursement clock for Shylock
onlyAntonio notSettled notDisbursed beforeDepositDeadline nonReentrant 
function depositCollateral() external payable ... {
    require(msg.value == collateralAmount,
            "Must deposit exactly twice the loan amount");
    collateralDeposited  = true;
    disbursementDeadline = block.timestamp + disbursementWindowSeconds;
    emit CollateralDeposited(msg.value, disbursementDeadline);
}Solidity

Scenario: Antonio agrees to the terms and locks his collateral. The exact amount is enforced not one wei more or less. The moment this succeeds, the contract holds collateralAmount ETH and Shylock's disbursement clock starts. The collateral will not move again until repayment or default.

disburseLoan()
Shylock sends the loan directly to Antonio starts the repayment clock
onlyShylock notSettled hasCollateral notDisbursed beforeDisbursementDeadline nonReentrant 
function disburseLoan() external payable ... {
    require(msg.value == loanAmount,
            "Must disburse exactly the loan amount");
    disbursed         = true;
    repaymentDeadline = block.timestamp + repaymentWindowSeconds;
    _safeTransfer(antonio, loanAmount);
    emit LoanDisbursed(loanAmount, repaymentDeadline);
}Solidity

Scenario: Shylock is satisfied that collateral is locked and makes the loan. He attaches exactly loanAmount ETH to this call. That ETH passes directly through to Antonio the contract never holds both the collateral and the loan simultaneously. Antonio's repayment clock starts the instant this transaction confirms.

The hasCollateral modifier ensures Shylock can never call this if Antonio has not deposited no loan can be made without the collateral already locked.

repay()
Antonio repays on-chain collateral returned and loan forwarded atomically
onlyAntonio notSettled isDisbursed beforeRepaymentDeadline nonReentrant 
function repay() external payable ... {
    require(msg.value == loanAmount,
            "Must repay exactly the loan amount");
    settled = true;                            // Effect first (CEI pattern)
    _safeTransfer(antonio, collateralAmount);  // Return collateral
    _safeTransfer(shylock, loanAmount);        // Forward repayment
    emit LoanRepaid(loanAmount, collateralAmount);
}Solidity

Scenario: Antonio repays before the deadline. He attaches exactly loanAmount ETH. The contract verifies the amount, returns his 2× collateral, and forwards the repayment to Shylock all in one atomic transaction. Either everything succeeds or nothing changes.

settled = true is written before any ETH moves. This is the Checks-Effects-Interactions pattern state is finalised before external calls are made, preventing any reentrancy exploit.

✓ Both parties made whole in one transaction: Antonio recovers 2× loan as collateral, Shylock recovers 1× loan as repayment. Net result both at zero, agreement honoured.
reclaimCollateral()
Antonio's exit if Shylock failed to disburse within the disbursement window
onlyAntonio notSettled hasCollateral notDisbursed afterDisbursementDeadline nonReentrant 
function reclaimCollateral() external ... {
    settled = true;
    _safeTransfer(antonio, collateralAmount);
    emit CollateralReclaimed(collateralAmount);
}Solidity

Scenario: Antonio deposited collateral in good faith but Shylock failed to disburse within his window. This function guarantees Antonio is never trapped his collateral has a defined exit regardless of Shylock's behaviour.

claimBond()
Antonio defaulted Shylock claims the full collateral as compensation
onlyShylock notSettled isDisbursed afterRepaymentDeadline nonReentrant 
function claimBond() external ... {
    settled = true;
    _safeTransfer(shylock, collateralAmount);
    emit BondForfeited(collateralAmount);
}Solidity

Scenario: Shylock disbursed the loan. The repayment deadline passed. Antonio never called repay(). Shylock claims the full collateral twice the loan amount as compensation. Antonio keeps the loan ETH as the cost of default.

The isDisbursed modifier is the logical guarantee: this can only fire if the loan was actually made. Shylock cannot claim a default on a loan he never disbursed. The collateral has been sitting in the contract since Antonio deposited it it is always there when this function fires.

✓ This is the wow moment. Antonio had the money. He had the function. He had the time. He chose not to repay. The contract enforced the penalty automatically. No court. No argument. The code ran.
cancel()
Shylock closes the contract if Antonio never deposited collateral
onlyShylock notSettled afterDepositDeadline nonReentrant 
function cancel() external ... {
    require(!collateralDeposited, "Collateral deposited");
    settled = true;
    emit ContractCancelled();
}Solidity

Scenario: Shylock deployed the contract but Antonio never deposited collateral and the deposit window closed. No ETH was ever locked so nothing moves. Shylock simply marks the contract settled to close it cleanly and emit a record of the cancellation.

12

View functions

function timeToDepositDeadline()      external view returns (uint256) { ... }
function timeToDisbursementDeadline() external view returns (uint256) { ... }
function timeToRepaymentDeadline()    external view returns (uint256) { ... }
function getBalance()                 external view returns (uint256) { ... }Solidity

view functions read state but never change it they cost no gas. All three countdown functions return 0 rather than producing an arithmetic error if the deadline has passed. Each also returns 0 if the preceding action has not yet occurred for example timeToRepaymentDeadline() returns 0 until the loan is disbursed.

getBalance() shows the contract's current ETH balance in wei. During the demo this confirms collateral is locked after deposit, shows it present throughout the repayment window, and confirms it transferred after claimBond().
